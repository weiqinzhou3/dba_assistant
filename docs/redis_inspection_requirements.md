# DBA Assistant Redis 巡检报告能力需求文档

## 1. 文档目的

本文档用于定义 DBA Assistant 下一阶段新增能力：**Redis 巡检报告能力**。  
该能力将在现有 RDB 分析能力基础上，进一步支持离线证据包巡检、在线 Redis 集群巡检、可选错误日志分析以及正式巡检报告输出，作为仓库后续 Phase 4 的主要交付目标。

## 2. 建设背景

当前项目已经完成 Redis RDB 分析主线，能够基于 prompt-first 模式完成本地 RDB、远程 RDB、MySQL-backed 分析与报告输出。现阶段需要继续建设 Redis 巡检能力，使助手不再局限于 RDB 文件分析，而能够对 Redis 集群运行状态、系统配置、复制关系、持久化、慢日志、错误日志、主机资源等进行综合巡检，并输出正式报告。

历史人工巡检材料显示，业务侧真实需要的交付形态具有以下特征：  
第一，巡检对象是“系统/集群”而非单节点；  
第二，报告要求正式、规范、适合对外或对客户交付；  
第三，分析不仅覆盖 Redis 数据库本身，也覆盖操作系统状态、主机资源、日志与高可用关系。

## 3. 建设目标

### 3.1 总体目标

开发新的 `redis_inspection_report` skill，使 DBA Assistant 可以基于统一 Deep Agent 架构，完成 Redis 巡检分析与正式报告输出。

### 3.2 分项目标

#### 3.2.1 正式化巡检报告输出
支持生成标准化、正式化、可交付的 Redis 巡检报告。报告不要求完全复刻历史样式，但应保留专业巡检报告的正式感、章节完整性和证据闭环。

#### 3.2.2 离线巡检能力
支持对历史采集证据包进行自动解析、集群归并、结构化分析和报告生成。

#### 3.2.3 在线巡检能力
支持根据 prompt 中提供的 Redis 信息自动连接集群，识别组织架构并采集巡检所需信息；当用户在 prompt 中额外提供 SSH 信息并要求分析 error log 时，支持一并分析近 N 天错误日志。

#### 3.2.4 Prompt-first 用户体验
保持 CLI 极薄原则。除密码等敏感信息外，不依赖 CLI 参数进行业务匹配与流程路由，而应由 LLM 理解 prompt 并拆解输入，再由 unified agent 选择对应 skill 与 tools 执行。

## 4. 范围定义

### 4.1 本期范围

本期建设范围包括以下内容：

1. Redis 巡检 skill 定义与接入。  
2. 离线证据包解析与集群归并。  
3. 在线 Redis 只读巡检采集。  
4. 可选 SSH 错误日志分析。  
5. 巡检统一分析器。  
6. 正式报告模板与 docx 输出。  
7. summary 输出与 docx 输出双模式支持。  
8. 审计与日志观测能力接入巡检流程。  

### 4.2 不在本期范围

以下内容不纳入本期交付：

1. Redis 写操作、自动修复、自动变更配置。  
2. 集群自动扩缩容、自动 failover。  
3. 非 Redis 中间件巡检。  
4. GUI 专用交互逻辑。  
5. 复杂多租户权限体系。  
6. 长周期趋势监控平台替代能力。  

## 5. 架构约束

本能力必须遵守当前仓库已经明确的统一执行形态：

`CLI / API / WebUI -> interface adapter -> one Deep Agent -> skills/tools`

因此必须满足以下约束：

### 5.1 CLI 极薄
CLI 仅作为 prompt 和少量敏感参数的载体，不负责 Redis 巡检模式选择、集群类型判断、章节选择、日志分析天数解析等业务逻辑。

### 5.2 Prompt-first
由 LLM 从用户 prompt 中识别并拆分下列信息：

- 是离线巡检还是在线巡检  
- 输入源是 tar 包、目录、Redis 地址还是 Redis + SSH  
- 是否需要分析 error log  
- 需要分析近多少天日志  
- 输出 summary 还是正式报告  
- 是否要求 docx  

### 5.3 Skill 驱动
由 unified Deep Agent 选择 `redis_inspection_report` skill 及其相关工具，不允许在 CLI 层硬编码 Phase 4 路由。

### 5.4 只读优先
在线巡检默认必须为只读路径。凡涉及潜在风险动作，必须通过后续 approval 机制显式扩展，不能在本阶段隐式放开。

### 5.5 报告链路复用
巡检报告必须复用现有 shared report pipeline，而不是另建一套与 RDB 分析平行的独立报告系统。

### 5.6 观测与审计复用
若仓库本地分支已经完成 Phase 5 的基础日志与审计能力，则 Phase 4 开发时**不再重复造一套新的日志/审计系统**。  
正确做法是：让 `redis_inspection_report` skill、离线归并、在线采集、日志分析、报告生成等流程统一接入已有 observability / audit baseline。  
也就是说，Phase 4 需要“接入并扩展现有日志与审计覆盖面”，而不是“跳过接入”或“平行重做一套”。

## 6. 输入路径设计

### 6.1 离线巡检输入

#### 6.1.1 输入形式
支持以下离线输入形式：

1. 单个 tar.gz 巡检证据包  
2. 某一目录下的多个节点采集包  
3. 已展开的采集目录  
4. 用户直接提供若干文件、目录、压缩包的混合输入  

#### 6.1.2 输入理解原则
离线输入路径应坚持 prompt-first。  
也就是说，用户给的未必一定是 tar 包，可能是目录、若干日志文件、某个节点采集结果、或者一组混合文件。  
系统应由 LLM 理解用户意图，然后由 Python 侧做**受控的能力判断与证据加载**：

- 不需要在 Python 中把所有自然语言表达都枚举写死。  
- 但必须实现一组明确支持的证据类型和加载器。  
- LLM 负责“这是离线巡检、这些路径应该作为证据输入”；  
- Python 负责“这些路径是否存在、属于哪类受支持证据、如何安全解析”。  

#### 6.1.3 当前样本特征
当前样本显示，离线材料往往不是按“集群维度”组织，而是按“节点采集结果”组织。因此，离线巡检的第一步必须是**证据归一化与集群归并**。

#### 6.1.4 离线路径目标
离线巡检必须能够：

1. 自动解析原始目录结构。  
2. 提取 IP、hostname、port、日期、日志文件、输出文件等元数据。  
3. 推断节点之间的主从关系、Cluster 关系或同系统归属关系。  
4. 将离散节点重组为“系统 -> 集群 -> 节点”的标准数据模型。  
5. 在标准模型基础上统一分析与报告。  

### 6.2 在线巡检输入

#### 6.2.1 输入形式
用户通过 prompt 提供 Redis 信息，例如：

- Redis 地址  
- 端口  
- 密码  
- 多个节点地址  

若需要错误日志分析，则额外在 prompt 中提供：

- SSH 主机  
- SSH 用户  
- SSH 密码  
- 日志时间范围要求，如“近 3 天”“近 7 天”  

#### 6.2.2 在线输入理解原则
在线巡检也坚持 prompt-first。  
用户可能只提供一个 Redis 地址、一个端口和一个密码；如果所有节点端口和密码一致，系统应默认以“种子节点发现”思路继续扩展，而不要求用户先把全量节点全部枚举出来。  
如果不同节点存在不同密码、不同端口、不同 SSH 登录方式，则允许用户在 prompt 中补充全量信息。

#### 6.2.3 在线路径目标
在线巡检必须能够：

1. 从一个或少量种子 Redis 节点出发，自动识别 Redis 拓扑组织。  
2. 判断是 Cluster、主从、哨兵还是单实例。  
3. 自动发现节点角色与复制关系。  
4. 采集 Redis 巡检所需信息。  
5. 在用户要求时通过 SSH 采集 error log 并做时间窗口分析。  

#### 6.2.4 在线发现原则
在线发现不应依赖“Python 里写死完整集群列表”，而应采用以下策略：

- 以用户提供的种子节点为入口。  
- 通过 `redis-cli info`、`INFO replication`、`CLUSTER NODES`、`CLUSTER INFO`、配置项与连接信息做节点发现。  
- 对于错误日志路径，不要求用户必须先给出完整日志文件位置；系统应优先通过 Redis 配置、服务信息、运行目录、常见日志落点和 SSH 只读探测做确定性定位。  
- 对于无法自动确定的情况，再通过 prompt 返回明确缺失项，而不是静默猜测。  

## 7. 核心数据模型

为保证离线和在线共用分析器，必须设计统一 inspection dataset。建议至少包含三层：

### 7.1 系统层
表示一个业务系统或一套巡检对象。

### 7.2 集群层
表示同一系统下的一套 Redis Cluster、主从集群或逻辑子集群。允许一个系统下存在多个集群。

### 7.3 节点层
表示具体 Redis 节点，至少包含：

- hostname  
- ip  
- port  
- role  
- version  
- source path  
- collect time  
- host facts  
- redis facts  
- log facts  

## 8. 功能需求

### 8.1 证据归一化

系统必须支持：

1. 解包 tar.gz。  
2. 识别节点采集目录结构。  
3. 提取节点元信息。  
4. 解析单节点巡检输出文件。  
5. 关联 `redis.log` 与对应节点。  
6. 构建标准 inspection dataset。  
7. 输出“集群归并结果”，用于调试与校验。  

### 8.2 集群归属识别

系统必须支持自动识别：

1. 同一 Redis Cluster 下的多个 master/slave 节点。  
2. 主从复制关系。  
3. 哨兵监控对象。  
4. 同一业务系统下的多子集群。  
5. 无法自动归并时给出可解释的待确认分组结果。  

### 8.3 Redis 巡检分析

Redis 层分析至少覆盖：

1. 架构检查  
2. 版本统一性检查  
3. OPS 检查  
4. 内存使用与碎片率检查  
5. `maxmemory` 与淘汰策略检查  
6. 持久化配置与状态检查  
7. key 空间状态检查  
8. 主从复制状态与延迟检查  
9. Cluster 状态检查  
10. slowlog 检查  
11. Redis 日志检查  
12. 配置合理性检查  

### 8.4 操作系统巡检分析

操作系统层分析至少覆盖：

1. OS 平台与内核版本  
2. `ulimit` 参数  
3. `iptables` / `selinux` 状态  
4. 系统负载  
5. 内存使用状态  
6. 透明大页状态  
7. swap 使用情况  
8. 磁盘使用情况  
9. 系统日志检查  

### 8.5 错误日志分析

当用户在 prompt 中明确要求分析 error log，且提供 SSH 信息时，系统应支持：

1. 只读采集 Redis error log。  
2. 支持按近 N 天窗口过滤。  
3. 汇总 error、warning、restart、fail、OOM、fork、AOF/RDB、replication 等事件。  
4. 输出节点维度与集群维度日志结论。  
5. 在报告中形成“错误日志与异常事件分析”章节。  

### 8.6 风险识别与整改建议

系统应针对巡检结果形成结构化问题项，每个问题项至少包含：

- 风险名称  
- 风险等级  
- 发现对象  
- 证据  
- 影响说明  
- 建议整改措施  

## 9. 报告需求

### 9.1 输出形式
必须支持：

1. summary 输出  
2. 正式 docx 报告输出  

### 9.2 报告风格
报告应满足以下要求：

1. 风格正式、专业、适合客户交付。  
2. 标题编号清晰。  
3. 先给结论，再给明细。  
4. 证据尽量内嵌，不只做口头描述。  
5. 集群维度和节点维度都要清晰。  
6. 保持较强的可读性与审计感。  

### 9.3 推荐章节结构
建议正式报告至少包含：

1. 巡检概述  
2. 巡检范围与输入说明  
3. 集群识别与架构总览  
4. 巡检结果总结  
5. 巡检目标及方法  
6. 系统配置检查  
7. 操作系统检查  
8. Redis 数据库检查  
9. 错误日志与异常事件分析  
10. 风险与整改建议  
11. 附录  

## 10. 交互与用户体验需求

### 10.1 Prompt-first 示例能力
用户应可以直接使用自然语言发起巡检，例如：

- 请根据这个离线巡检包生成正式 Redis 巡检报告。  
- 请连接这套 Redis Cluster 做巡检，并输出 docx 报告。  
- 请连接 Redis 并分析近 7 天 error log。  
- 请基于离线证据包，识别有哪些集群，并按集群出巡检总结。  

系统应由 LLM 自动识别输入意图与所需路径。

### 10.2 CLI 约束
CLI 不应要求用户显式输入如下业务参数：

- cluster type  
- inspection mode  
- report chapter selection  
- log day window  
- cluster grouping mode  

这些应由 LLM 从 prompt 中推断。

## 11. 日志与审计需求

巡检 skill 必须接入统一 observability / audit baseline。  
至少需要记录：

1. 本次执行的输入模式  
2. 识别到的系统数 / 集群数 / 节点数  
3. 离线解包与归并阶段  
4. 在线采集阶段  
5. Redis 检查阶段  
6. OS 检查阶段  
7. error log 分析阶段  
8. 报告生成阶段  
9. 成功 / 失败 / 部分完成状态  

## 12. 分阶段实施建议

### 12.1 第一阶段：离线巡检先打通
交付目标：

1. 能读 tar 包或目录。  
2. 能自动归并系统/集群/节点。  
3. 能基于离线证据生成正式报告。  
4. 能以历史报告为基准做结果比对。  

### 12.2 第二阶段：在线巡检接入
交付目标：

1. 根据 Redis 信息自动连通巡检。  
2. 自动识别拓扑组织。  
3. 支持 SSH 采集 error log。  
4. 与离线分析器和报告器复用。  

### 12.3 第三阶段：规则沉淀与模板优化
交付目标：

1. 补齐更多架构类型适配。  
2. 优化风险评级规则。  
3. 优化正式报告模板。  
4. 提升对复杂目录和复杂集群的归并准确率。  

## 13. 验收标准

本需求文档对应的首版交付，至少应满足以下验收标准：

1. 能够对离线巡检包完成解析和集群归并。  
2. 能够生成一份正式的 Redis 巡检 docx 报告。  
3. 报告能够按系统/集群组织，而不是按文件散乱输出。  
4. 在线模式下，能够根据 prompt 中 Redis 信息自动完成巡检。  
5. 当用户提供 SSH 信息并要求分析 error log 时，能够分析近 N 天错误日志。  
6. CLI 不需要额外承担业务匹配职责。  
7. 整个流程保持 prompt-first 与 unified Deep Agent 架构一致。  
8. 巡检流程接入现有 observability / audit baseline，而不是另造一套。  

## 14. 针对设计问题的明确答复

### 14.1 关于 Phase 5 已完成后如何开发 Phase 4
如果你的本地分支已经完成了 Phase 5 的基础日志和审计能力，那么开发 Phase 4 时**仍然要把巡检流程接入日志和审计体系**。  
但这不意味着重新开发一遍 Phase 5。正确做法是：

- 复用已经做好的日志初始化、execution context、audit event schema。  
- 在离线归并、在线采集、错误日志分析、报告生成等新阶段上补充覆盖。  
- 保持统一 observability baseline，不要为 inspection 单独再造平行日志系统。  

结论：**要接入，但不要重做。**

### 14.2 关于离线输入不一定是 tar 包
是的。  
系统应以 LLM 理解用户意图为主，而不是在 Python 中把所有自然语言情况写死。  
但 Python 侧仍然需要提供明确支持的证据类型与加载器，负责安全解析与校验。  

结论：**不需要把所有用户说法都写死，但需要有受控的离线证据加载能力。**

### 14.3 关于在线分析如何扩展到整个集群及错误日志
是的。  
在线巡检必须支持从一个种子 Redis 节点出发，自动识别整个集群或主从拓扑，而不是要求用户总是先给全量节点。  
同时，对于错误日志位置，也不应该依赖用户总是先告诉你完整路径；系统应优先通过 Redis 配置、服务信息、运行目录和 SSH 只读探测来定位。  

结论：  
**种子节点发现 + 拓扑扩展 + 受控日志定位** 应作为在线巡检的基础能力。

## 15. 结论

当前项目可以正式从“RDB 分析”进入“Redis 巡检报告”阶段。  
本阶段的本质不是新增一个零散工具，而是交付一个新的 `redis_inspection_report` skill，支持离线证据包巡检与在线只读巡检两条路径，并输出正式、标准化、可交付的 Redis 巡检报告。

同时，本阶段的关键架构基线已经明确：

1. CLI 保持极薄。  
2. 业务理解与 inputs 拆分主要由 LLM 完成。  
3. 离线与在线尽量共用统一分析器与报告器。  
4. 巡检流程必须复用现有日志与审计基线，而不是另起一套。
